# Day 62: Result/Either Patterns for Errors

## 📝 Today's Tasks

1.  **Result Type:** Implement `Ok<T> | Err<E>` as a discriminated union.
2.  **map/andThen:** Add helpers for transforming success values.
3.  **Replace Throws:** Convert a throw-based flow to Result-returning functions.
4.  **Interop:** Bridge Result ↔ exceptions at boundaries.

## 📖 Task Explanations

Result types make errors visible in signatures. Ideal for domain logic; exceptions may still appear at process edges. Keep one strategy per layer.

## 💡 What You'll Gain

A practical Result type used to model recoverable failures explicitly.

## 📚 Dictionary for Developers

*   **Result/Either:** A union representing success or failure values.
*   **Recoverable Error:** A failure callers are expected to handle.
*   **Boundary Bridging:** Converting between Result and exceptions.
