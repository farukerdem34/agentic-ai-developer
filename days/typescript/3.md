# Day 3: Primitive Types, null, and Top/Bottom Types

## 📝 Today's Tasks

1.  **Primitives Drill:** Practice `string`, `number`, `boolean`, `bigint`, and `symbol`.
2.  **Enable Null Safety:** Turn on `strictNullChecks` (via `strict`) and handle `null`/`undefined` deliberately.
3.  **any vs unknown:** Write examples showing why `unknown` requires narrowing while `any` disables checking.
4.  **void and never:** Implement a logging function returning `void` and an always-throwing helper typed as `never`.

## 📖 Task Explanations

Primitives mirror JavaScript values, but nullability is stricter under `strictNullChecks`. Prefer `unknown` over `any` when data arrives untrusted. `void` means no useful return value; `never` means a function does not complete normally—critical for exhaustiveness checking later.

## 💡 What You'll Gain

Precise null-safe primitive usage and a clear mental model for `any`, `unknown`, `void`, and `never`.

## 📚 Dictionary for Developers

*   **strictNullChecks:** Treats `null`/`undefined` as separate and not assignable to other types by default.
*   **`any`:** Disables type checking for a value—avoid in production.
*   **`unknown`:** Safe top type that must be narrowed before use.
*   **`never`:** Bottom type for values that never occur / unreachable code.
