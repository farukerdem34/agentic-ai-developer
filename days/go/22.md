# Day 22: Table-Driven Tests

## 📝 Today's Tasks

1.  **Design a Table:** Create a slice of cases with inputs and expected outputs.
2.  **Subtests:** Use `t.Run` for each case name.
3.  **Edge Cases:** Include empty input, errors, and boundary values.
4.  **Refactor for Testability:** Adjust production code to make cases easy to express.

## 📖 Task Explanations

Table-driven tests are the Go standard for covering many scenarios without duplication. Named subtests improve failure diagnosis in CI.

## 💡 What You'll Gain

You will write expressive table-driven tests that cover happy paths and edge cases.

## 📚 Dictionary for Developers

*   **Table-Driven Test:** A test that iterates a list of cases with inputs and expected results.
*   **Subtest:** A named nested test created with `t.Run`.
*   **Edge Case:** An unusual but valid input at a boundary of behavior.
