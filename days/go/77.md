# Day 77: Message Brokers Intro

## 📝 Today's Tasks

1.  **Why Messaging:** Decouple producers/consumers and smooth load spikes.
2.  **Pick a Starter:** Experiment with NATS, RabbitMQ, or Kafka conceptually; implement a simple queue locally.
3.  **Publish/Subscribe:** Emit an event when a domain action occurs.
4.  **At-Least-Once Reality:** Design consumers to be idempotent.

## 📖 Task Explanations

Messaging enables asynchronous workflows. Assume duplicates; make handlers idempotent.

## 💡 What You'll Gain

A basic producer/consumer path and an idempotent consumer mindset.

## 📚 Dictionary for Developers

*   **Message Broker:** Infrastructure that routes messages between services.
*   **Pub/Sub:** Publishers emit events; subscribers receive them.
*   **At-Least-Once Delivery:** Messages may be delivered more than once.
