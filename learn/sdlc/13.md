## Day 13

### Environment Map

| Environment | Purpose | Who Uses It |
|-------------|---------|-------------|
| Local | Development and testing on developer machine | Developers |
| Staging | Pre-production validation, realistic testing | QA, limited users |
| Production | Live version, end-users | All users |

### Promotion Path

| Step | Action |
|------|--------|
| 1 | Developer commits code locally and tests |
| 2 | Developer pushes to `staging` branch |
| 3 | CI runs tests, builds artifact |
| 4 | Artifact deploys to Staging environment |
| 5 | QA validates in Staging |
| 6 | Artifact promotes to Production |

### Config Awareness

| # | Config Item | Why It Differs |
|---|-------------|----------------|
| 1 | API Keys | Test keys for local/staging, production keys for prod |
| 2 | Database URLs | Different databases per environment |
| 3 | Feature Flags | Enable/disable features for testing |
| 4 | Logging Level | Debug in local, minimal in production |

### Unsafe Shortcut

**Why it's risky:**
- Untested code can introduce new bugs
- No validation in realistic environment
- Potential data loss or downtime
- No rollback plan

**Safer alternative:**
Even for critical hotfixes, run at minimum:
1. Unit tests locally
2. Deploy to Staging (even briefly)
3. Smoke test critical paths
4. Deploy to Production with feature flag
5. Monitor for errors

**Blue-green deployment** allows instant rollback by switching traffic between two identical environments.
