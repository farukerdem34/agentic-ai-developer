## Day 14

### Maintenance Types

| Type | Definition | Example |
|------|------------|---------|
| Corrective | Fixes existing bugs, crashes, or failures | Tool calling not working properly — must be fixed ASAP |
| Adaptive | Adjusts software to changes in environment | Changing LLM model for performance and correct information |
| Perfective | Upgrades functionality, UX, and performance | Improving UI speed, LLM tool calling performance improvements |
| Preventive | Reduces future risks before problems occur | Estimating bugs on tool calling and modifying code |

### Monitoring Signals

| # | Signal |
|---|--------|
| 1 | Time to first token |
| 2 | Error logs, rates, endpoints |
| 3 | Request count per endpoint |
| 4 | RAM and CPU usage (time series data) |
| 5 | Audit logs |

### Incident Timeline

| Step | Action |
|------|--------|
| 1 | **Detect** — Critical error log notification received |
| 2 | **Investigate** — Determine root cause (e.g., Tool Router failure) |
| 3 | **Mitigate** — Disable feature flag for tool calling |
| 4 | **Fix** — Examine bug, deploy fix to staging |
| 5 | **Review** — Post-incident review, document findings |

### On-call Empathy

| # | Information |
|---|-------------|
| 1 | When it happened (timestamp) |
| 2 | Who triggered it (user ID, request ID) |
| 3 | Which component failed (Tool Router, LLM API, etc.) |
| 4 | What is the impact (error rate, affected users) |
| 5 | Recent changes (deployments, config changes) |
